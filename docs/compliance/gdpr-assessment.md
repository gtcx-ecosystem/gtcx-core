# GDPR Data Protection Assessment — gtcx-core

**Document ID:** GTCX-CORE-GDPR-001
**Version:** 1.0
**Date:** 2026-05-08
**Status:** Active
**Review cycle:** Annually or on scope change

---

## 1. Scope

gtcx-core is a **cryptographic library** distributed via npm and crates.io. It executes entirely within the consuming application's process. It has no network listeners, no database connections, no user accounts, and no telemetry.

---

## 2. Personal Data Inventory

| Category                   | Data Processed | Data Stored | Data Transmitted |
| -------------------------- | -------------- | ----------- | ---------------- |
| Names / identifiers        | None           | None        | None             |
| Contact information        | None           | None        | None             |
| Biometric data             | None           | None        | None             |
| Location data              | None           | None        | None             |
| Financial data             | None           | None        | None             |
| Authentication credentials | None           | None        | None             |

**Conclusion:** gtcx-core processes **zero personal data** as defined by GDPR Art. 4(1).

Cryptographic key material (Ed25519 private keys, DIDs) generated by the library is not personal data in itself — it becomes personal data only when a downstream service binds it to an identifiable natural person. That binding is the downstream service's responsibility.

---

## 3. DPIA Determination

Per GDPR Art. 35(1), a Data Protection Impact Assessment is required when processing is "likely to result in a high risk to the rights and freedoms of natural persons."

**Determination: DPIA is not required for gtcx-core.**

Rationale:

- The library processes no personal data
- It performs no profiling, scoring, or automated decision-making
- It has no access to data subjects
- It operates entirely within the caller's memory space

Downstream services that use gtcx-core to process personal data (e.g., binding DIDs to real identities) **must conduct their own DPIA**.

---

## 4. Privacy by Design (Art. 25)

Although gtcx-core processes no personal data, its design supports downstream privacy compliance:

| Principle                   | Implementation                                                                      | Evidence                                                             |
| --------------------------- | ----------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| Data minimization           | No telemetry, no analytics, no usage data collection                                | No network calls in source code                                      |
| Purpose limitation          | Cryptographic operations only — no data interpretation                              | Package scope defined in specs                                       |
| Storage limitation          | Key material zeroized after use via `secureWipe()` (TS) and `Zeroize` derive (Rust) | `packages/crypto/src/signing.ts`, `rust/gtcx-crypto/src/keys/mod.rs` |
| Integrity & confidentiality | Ed25519 signing, AES-256-GCM encryption, constant-time comparison                   | `docs/security/security-framework.md`                                |
| Pseudonymization            | DID-based identity (hash-derived, not PII-linked)                                   | `packages/identity/src/`                                             |

---

## 5. Security of Processing (Art. 32)

| Measure          | Implementation                                                 |
| ---------------- | -------------------------------------------------------------- |
| Encryption       | AES-256-GCM for storage, Ed25519 for signing                   |
| Pseudonymization | DIDs are cryptographic identifiers, not PII                    |
| Confidentiality  | Private keys never logged; `redactSecrets()` default sanitizer |
| Integrity        | SHA-256/BLAKE3 hash verification, hash-chained audit trails    |
| Availability     | Offline-first design with conflict resolution                  |
| Resilience       | 5 sync conflict strategies, circuit breaker in API client      |

---

## 6. Data Subject Rights

Not applicable. gtcx-core has no data subjects, no user accounts, and no stored personal data.

Downstream services implementing data subject rights (access, erasure, portability) should note:

- DID deletion: `@gtcx/identity` supports DID revocation
- Key destruction: `secureWipe()` and `Zeroize` ensure cryptographic erasure
- Audit trail: Hash-chained logs can be marked as "erasure-requested" while maintaining chain integrity for regulatory retention

---

## 7. Cross-Border Transfer

Not applicable. gtcx-core is distributed as open-source software via npm/crates.io. No data crosses borders through the library.

---

## 8. Downstream Obligations

Any service using gtcx-core that processes personal data must:

1. Conduct its own DPIA for its specific processing activities
2. Establish a legal basis for processing (Art. 6)
3. Implement its own data subject rights mechanisms
4. Maintain its own records of processing activities (Art. 30)
5. Appoint a DPO if required by Art. 37

gtcx-core provides the cryptographic primitives. Privacy compliance is an application-level concern.
